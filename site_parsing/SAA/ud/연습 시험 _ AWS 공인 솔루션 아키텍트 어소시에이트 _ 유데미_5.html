질문 1:
건너뜀

us-west-1a 가용 영역에서 방금 인스턴스를 종료했습니다. 이제 연결된 EBS 볼륨을 다른 인스턴스에 연결할 수 있습니다. 인턴이 us-west-1b 가용 영역에서 새 Linux EC2 인스턴스를 시작하고 EBS 볼륨 연결을 시도하고 있습니다. 인턴은 불가능하며 귀하의 도움이 필요하다고 알려줍니다.

다음 중 어떤 설명을 그들에게 제공하시겠습니까?

설명

올바른 옵션:

EBS 볼륨은 AZ로 잠겨 있습니다.

Amazon EBS 볼륨은 인스턴스에 연결할 수 있는 내구성 있는 블록 수준 스토리지 디바이스입니다. 볼륨을 인스턴스에 연결한 후에는 물리적 하드 드라이브를 사용하는 것처럼 볼륨을 사용할 수 있습니다. EBS 볼륨은 유연합니다. 현재 세대 인스턴스 유형에 연결된 현재 세대 볼륨의 경우 동적으로 크기를 늘리고, 프로비저닝된 IOPS 용량을 수정하고, 라이브 프로덕션 볼륨에서 볼륨 유형을 변경할 수 있습니다.

EBS 볼륨을 생성하면 단일 하드웨어 구성 요소의 장애로 인한 데이터 손실을 방지하기 위해 가용 영역 내에서 자동으로 복제됩니다. 동일한 가용 영역에 있는 EC2 인스턴스에 EBS 볼륨을 연결할 수 있습니다.

잘못된 옵션:

EBS 볼륨은 리전 잠금 - 리전이 아닌 가용 영역으로 제한됩니다.

필요한 IAM 권한이 없습니다 . - 이것도 가능성이 있지만 권한이 문제가 아닌 경우 여전히 가용성 영역으로 제한됩니다.

EBS 볼륨이 암호화됨 - 이것은 EBS 볼륨을 연결하는 기능에 영향을 미치지 않습니다.

참조:

https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/EBSVolumes.html

질문 2:
건너뜀

회사의 클라우드 설계자는 Route 53을 사용하여 ALB(Application Load Balancer)를 가리키는 도메인이 있는 기본 웹 사이트에 대한 DNS 레코드를 구성하는 솔루션을 설정했습니다. 회사는 기본 웹 사이트를 사용할 수 없는 경우 백업으로 구성된 정적 오류 페이지로 사용자를 안내하는 솔루션을 원합니다.

변경 사항을 최소한으로 유지하면서 회사의 요구 사항을 충족하는 구성은 무엇입니까?

설명

올바른 옵션:

장애 조치 라우팅 정책의 Route 53 활성-수동 유형을 설정합니다. Route 53 상태 확인에서 ALB 엔드포인트가 비정상이라고 판단하면 트래픽이 Amazon S3 버킷에서 호스팅되는 정적 오류 페이지로 전환됩니다.

대부분의 시간 동안 기본 리소스 또는 리소스 그룹을 사용할 수 있고 모든 기본 리소스를 사용할 수 없게 된 경우 보조 리소스 또는 리소스 그룹을 대기 상태로 두려는 경우 활성-수동 장애 조치 구성을 사용합니다. 쿼리에 응답할 때 Route 53에는 정상 기본 리소스만 포함됩니다. 모든 기본 리소스가 비정상이면 Route 53은 DNS 쿼리에 대한 응답으로 정상 보조 리소스만 포함하기 시작합니다.

잘못된 옵션:

장애 조치 라우팅 정책의 Route 53 활성-활성 유형을 설정합니다. Route 53 상태 확인에서 ALB 엔드포인트가 비정상이라고 판단하면 트래픽이 Amazon S3 버킷에서 호스팅되는 정적 오류 페이지로 전환됩니다.- Route 53에는 활성-활성 장애 조치 라우팅 정책과 같은 것이 없기 때문에 이 옵션이 선택 항목으로 추가되었습니다. 장애 조치 라우팅 정책 이외의 모든 라우팅 정책(또는 라우팅 정책 조합)을 사용하여 활성-활성 장애 조치를 구성할 수 있습니다. 장애 조치 라우팅 정책만 사용하여 능동-수동 장애 조치를 구성합니다. 활성-활성 장애 조치 구성에서 동일한 이름, 동일한 유형(예: A 또는 AAAA) 및 동일한 라우팅 정책(예: 가중 또는 대기 시간)이 있는 모든 레코드는 Route 53에서 비정상으로 간주하지 않는 한 활성 상태입니다. Route 53은 정상적인 레코드를 사용하여 DNS 쿼리에 응답할 수 있습니다.

Route 53 지연 시간 기반 라우팅을 사용합니다. 표시할 오류 페이지가 있는 Amazon S3 버킷을 가리키는 지연 시간 레코드 생성 - 애플리케이션이 여러 AWS 리전에서 호스팅되는 경우 지연 시간이 가장 짧은 AWS 리전에서 요청을 처리하여 사용자의 성능을 개선할 수 있습니다. - 지연 시간 기반 라우팅이며 현재 사용 사례에 도움이 되지 않습니다.

표시할 오류 페이지가 있는 Amazon S3 버킷에 최소 가중치를 부여하려면 Route 53 가중 라우팅을 사용하십시오. 기본 실패의 경우 요청이 오류 페이지로 라우팅됩니다. 가중치 기반 라우팅을 사용하면 여러 리소스를 단일 도메인 이름(example.com) 또는 하위 도메인 이름(acme.example.com)과 연결하고 라우팅할 트래픽 양을 선택할 수 있습니다. 각 자원. 이는 로드 밸런싱 및 새 버전의 소프트웨어 테스트를 비롯한 다양한 목적에 유용할 수 있습니다. 이것은 현재 사용 사례에 유용하지 않습니다.

질문 3:
건너뜀

솔루션 아키텍트는 회사의 엔지니어링 팀과 협력하여 서버리스 아키텍처를 사용하여 REST API를 생성하도록 고용되었습니다.

다음 중 회사를 서버리스 아키텍처로 전환하기 위해 권장하는 솔루션은 무엇입니까?

설명

올바른 옵션:

Lambda 기능을 노출하는 API Gateway

Amazon API Gateway는 개발자가 모든 규모에서 API를 쉽게 생성, 게시, 유지 관리, 모니터링 및 보호할 수 있게 해주는 완전관리형 서비스입니다. API는 애플리케이션이 백엔드 서비스의 데이터, 비즈니스 로직 또는 기능에 액세스하기 위한 "정문" 역할을 합니다.

API 게이트웨이 작동 방식: - https://aws.amazon.com/api-gateway/ 를 통해

AWS Lambda를 사용하면 서버를 프로비저닝하거나 관리하지 않고도 코드를 실행할 수 있습니다. 사용한 컴퓨팅 시간에 대해서만 비용을 지불합니다.

Lambda 함수 작동 방식: - https://aws.amazon.com/lambda/ 를 통해

API Gateway는 RESTful API를 통해 Lambda 기능을 노출할 수 있습니다. 둘 다 AWS에서 제공하는 서버리스 옵션이므로 제공하는 모든 기능을 고려할 때 이 시나리오에 적합한 선택입니다.

잘못된 옵션:

전면에 Lambda가 있는 Fargate - Lambda는 RESTful API 요청을 직접 처리할 수 없습니다. Amazon API Gateway를 사용하여 사용자 지정 RESTful API를 정의하여 HTTPS를 통해 Lambda 함수를 호출할 수 있습니다. 따라서 전면 서비스로 Lambda를 사용하는 Fargate는 잘못된 조합이지만 Fargate와 Lambda는 모두 서버리스입니다.

Amazon EC2의 ECS를 사용하는 공용 Application Load Balancer - Amazon EC2의 ECS는 서버리스에 포함되지 않으므로 이 사용 사례에서 고려할 수 없습니다.

EC2를 백엔드로 사용하는 Route 53 - Amazon EC2는 서버리스 서비스가 아니므로 이 사용 사례에서 고려할 수 없습니다.

질문 4:
건너뜀

대학의 컴퓨터 비전 연구원은 EC2 인스턴스에서 실행되는 독점 알고리즘에 대한 I/O 바운드 프로세스를 최적화하려고 합니다. 이상적인 스토리지는 결과를 Amazon S3에 다시 업로드하기 전에 임시 스토리지 공간에서 파일 처리를 수행할 때 고성능 IOPS를 용이하게 합니다.

솔루션 아키텍트로서 다음 AWS 스토리지 옵션 중 가장 성능이 뛰어나고 비용이 최적인 것으로 추천하는 것은 무엇입니까?

설명

올바른 옵션:

스토리지 유형으로 인스턴스 스토어와 함께 EC2 인스턴스 사용

인스턴스 스토어는 인스턴스에 대한 임시 블록 수준 스토리지를 제공합니다. 이 저장소는 호스트 컴퓨터에 물리적으로 연결된 디스크에 있습니다. 인스턴스 스토어는 버퍼, 캐시, 스크래치 데이터 및 기타 임시 콘텐츠와 같이 자주 변경되는 정보의 임시 스토리지 또는 로드 밸런싱된 웹 서버 풀과 같은 인스턴스 집합 전체에 복제되는 데이터에 이상적입니다. . 일부 인스턴스 유형은 NVMe 또는 SATA 기반 SSD(Solid-State Drive)를 사용하여 높은 임의 I/O 성능을 제공합니다. 이는 대기 시간이 매우 짧은 스토리지가 필요하지만 인스턴스가 종료될 때 데이터를 유지할 필요가 없거나 내결함성 아키텍처를 활용할 수 있는 경우에 좋은 옵션입니다.

Instance Store는 높은 임의 I/O 성능을 제공하므로 임시 저장 공간 역할을 할 수 있으며 이러한 볼륨은 인스턴스 사용 비용의 일부로 포함되므로 이것이 올바른 옵션입니다.

https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/InstanceStorage.html 을 통한 Amazon EC2 인스턴스 스토어

잘못된 옵션:

스토리지 옵션으로 EBS 범용 SSD(gp2)와 함께 EC2 인스턴스 사용 - 범용 SSD(gp2) 볼륨은 광범위한 워크로드에 이상적인 비용 효율적인 스토리지를 제공합니다. 이러한 볼륨은 한 자릿수 밀리초의 지연 시간과 확장된 기간 동안 3,000 IOPS까지 버스트할 수 있는 기능을 제공합니다. 최소 100 IOPS(33.33 GiB 이하)와 최대 16,000 IOPS(5,334 GiB 이상) 사이에서 기본 성능은 GiB 볼륨 크기당 3 IOPS로 선형적으로 확장됩니다. AWS는 99%의 시간 동안 프로비저닝된 성능을 제공하도록 gp2 볼륨을 설계합니다. gp2 볼륨의 크기 범위는 1GiB~16TiB입니다. EBS gp2는 영구 스토리지이며 인스턴스 스토어보다 비용이 많이 들기 때문에(스토리지 볼륨 비용이 EC2 인스턴스 비용에 추가됨) 이 옵션은 올바르지 않습니다.

스토리지 옵션으로 EBS 프로비저닝된 IOPS SSD(io1)와 함께 EC2 인스턴스 사용 - 프로비저닝된 IOPS SSD(io1) 볼륨은 I/O 집약적인 워크로드, 특히 스토리지 성능과 일관성에 민감한 데이터베이스 워크로드의 요구 사항을 충족하도록 설계되었습니다. 버킷 및 크레딧 모델을 사용하여 성능을 계산하는 gp2와 달리 io1 볼륨을 사용하면 볼륨을 생성할 때 일관된 IOPS 속도를 지정할 수 있으며 Amazon EBS는 99.9%의 시간 동안 프로비저닝된 성능을 제공합니다. EBS io1은 영구 스토리지이며 인스턴스 스토어보다 비용이 많이 들기 때문에(스토리지 볼륨 비용이 EC2 인스턴스 비용에 추가됨) 이 옵션은 올바르지 않습니다.

스토리지 옵션으로 EBS 처리량 최적화 HDD(st1)와 함께 EC2 인스턴스 사용 - 처리량 최적화 HDD(st1)는 빅 데이터 및 데이터 웨어하우스와 같이 자주 액세스하고 처리량 집약적인 워크로드를 위해 설계된 저비용 HDD 볼륨입니다. EBS st1은 영구 스토리지이며 인스턴스 스토어보다 비용이 많이 들기 때문에(스토리지 볼륨 비용이 EC2 인스턴스 비용에 추가됨) 이 옵션은 올바르지 않습니다.

질문 5:
건너뜀

며칠 전 새로운 Auto Scaling 그룹을 배포한 후 한 회사에서 애플리케이션 성능이 저하되었음을 알아차렸습니다. 조사 결과 팀은 Auto Scaling 그룹에 대해 선택한 시작 구성이 애플리케이션 워크플로를 처리하도록 최적화되지 않은 잘못된 인스턴스 유형을 사용하고 있음을 발견했습니다.

솔루션 아키텍트로서 이 문제에 대한 장기적인 해결책을 제공하기 위해 무엇을 추천하시겠습니까?

설명

올바른 옵션:

올바른 인스턴스 유형을 사용하려면 새 시작 구성을 생성하십시오. 이 새로운 시작 구성을 사용하도록 Auto Scaling 그룹을 수정합니다. 더 이상 필요하지 않은 이전 시작 구성을 삭제합니다.

시작 구성은 Auto Scaling 그룹이 EC2 인스턴스를 시작하는 데 사용하는 인스턴스 구성 템플릿입니다. 시작 구성을 생성할 때 인스턴스에 대한 정보를 지정합니다. Amazon 머신 이미지(AMI)의 ID, 인스턴스 유형, 키 쌍, 하나 이상의 보안 그룹 및 블록 디바이스 매핑을 포함합니다.

일단 생성된 시작 구성은 수정할 수 없습니다. 올바른 옵션은 올바른 인스턴스 유형을 사용하도록 새 시작 구성을 생성하는 것입니다. 그런 다음 이 새로운 시작 구성을 사용하도록 Auto Scaling 그룹을 수정합니다. 마지막으로 정리하려면 더 이상 필요하지 않은 이전 시작 구성을 삭제하십시오.

잘못된 옵션:

대부분의 데이터를 디스크나 SSD에 저장하는 PostgreSQL, Cassandra, MongoDB 및 기타 데이터베이스와 달리 모든 Redis 데이터는 서버의 주 메모리에 상주합니다. 대부분의 작업에 디스크 왕복이 필요한 기존의 디스크 기반 데이터베이스와 비교할 때 Redis와 같은 메모리 내 데이터 저장소는 동일한 페널티를 받지 않습니다. 따라서 훨씬 더 많은 작업과 더 빠른 응답 시간을 지원할 수 있습니다. 그 결과 평균 읽기 또는 쓰기 작업이 1밀리초도 안 걸리고 초당 수백만 건의 작업을 지원하는 놀랍도록 빠른 성능을 제공합니다.

Redis에는 대규모 실시간 지리 공간 데이터 작업을 위한 특수 목적 명령이 있습니다. 두 요소(예: 사람 또는 장소) 사이의 거리 찾기 및 점의 지정된 거리 내에 있는 모든 요소 찾기와 같은 작업을 수행할 수 있습니다.
